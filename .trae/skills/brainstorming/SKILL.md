# Brainstorming - 需求分析与规划

## 核心目标

在开始编码之前，确保：
1. 需求清晰、可执行
2. 技术方案合理
3. 潜在问题已识别
4. 开发计划明确

## 使用场景

### 必须调用的场景
- 用户提出新功能开发需求
- 用户发送需求文档需要分析
- 用户规划新版本功能
- 需求模糊或存在多种理解
- 复杂功能需要拆解

### 不需要调用的场景
- 简单的 bug 修复
- 明确的代码修改（如"把这个值从 5 改成 10"）
- 已经有详细文档且无疑问

## 工作流程

### 第一阶段：需求理解

1. **仔细阅读需求文档**（如果有）
   - 标记不清晰的点
   - 识别核心功能点
   - 找出依赖关系

2. **主动提问澄清**
   - 目标用户是谁？
   - 使用场景是什么？
   - 优先级如何？
   - 有什么限制条件？

3. **总结确认**
   - 复述理解的需求
   - 确认关键点
   - 识别遗漏

### 第二阶段：方案设计

1. **技术选型**
   - 需要新增哪些文件？
   - 需要修改哪些现有模块？
   - 是否需要第三方库？

2. **架构设计**
   - 数据流如何设计？
   - 组件如何划分？
   - 状态如何管理？

3. **风险评估**
   - 技术难点在哪里？
   - 有什么潜在坑？
   - 如何规避风险？

### 第三阶段：开发规划

1. **任务拆解**
   - 将大功能拆成小任务
   - 每个任务可独立验证
   - 估算工作量

2. **优先级排序**
   - 核心功能优先
   - 依赖关系考虑
   - 快速验证优先

3. **验收标准**
   - 每个任务如何验证完成？
   - 功能完成的定义是什么？

## 输出模板

### 需求分析文档

```markdown
# 需求分析 - [功能名称]

## 一、需求概述
- 目标：[解决什么问题]
- 用户：[谁使用]
- 场景：[在什么情况下使用]

## 二、功能列表
### P0（必须）
- [ ] 功能点 1
- [ ] 功能点 2

### P1（重要）
- [ ] 功能点 3

### P2（可选）
- [ ] 功能点 4

## 三、技术方案
### 新增文件
- `path/to/new/file.js` - 用途说明

### 修改文件
- `path/to/modify/file.js` - 修改内容

### 数据结构
```javascript
{
  // 数据格式定义
}
```

## 四、开发计划
1. [ ] 任务 1 - 预计耗时
2. [ ] 任务 2 - 预计耗时
3. [ ] 任务 3 - 预计耗时

## 五、风险与注意事项
- 风险点 1：[描述 + 应对方案]
- 风险点 2：[描述 + 应对方案]
```

## 关键原则

### 1. 不要急于编码
- 先理解，再设计，最后编码
- 模糊的需求必然导致返工
- 多问问题不丢人

### 2. 保持批判性思维
- 质疑需求的合理性
- 提出更好的替代方案
- 考虑边界情况

### 3. 文档化
- 讨论结果要记录
- 方案要可视化（流程图、时序图）
- 决策要有依据

### 4. 用户确认
- 每个阶段都要用户确认
- 不要假设用户理解技术术语
- 用用户语言沟通

## 常用问题清单

### 需求澄清
- 这个功能主要解决什么问题？
- 目标用户是谁？他们现在怎么解决这个问题？
- 使用频率如何？
- 有没有类似的参考产品？

### 技术评估
- 现有架构是否支持？
- 需要哪些第三方依赖？
- 性能要求是什么？
- 数据安全如何保证？

### 优先级
- 最核心不能少的功能是什么？
- 哪些功能可以后续迭代？
- 有没有时间限制？

## 示例对话

**用户**: "我准备做 1.3.0 版本，需求文档发给你"

**你**: 
"好的，我会仔细阅读需求文档。在开始分析之前，我想先了解几个问题：

1. 1.3.0 版本的核心目标是什么？
2. 这些功能中，哪些是必须在这个版本完成的？
3. 有没有时间上的限制？

我先阅读文档，然后会输出一份详细的需求分析和技术方案，我们再一起讨论确认。"

---

阅读后：

"我已经看完需求文档，总结如下：

**核心功能**（我理解的）：
1. ...
2. ...

**几个需要确认的点**：
1. [疑问 1]
2. [疑问 2]

**技术难点预判**：
1. ...

我的理解对吗？有没有遗漏或理解偏差的地方？"

## 注意事项

1. **不要一次性问太多问题** - 分批问，等用户回答
2. **不要假设** - 不确定的就问
3. **不要跳过确认环节** - 每个阶段都要确认
4. **保持灵活** - 方案可以调整
