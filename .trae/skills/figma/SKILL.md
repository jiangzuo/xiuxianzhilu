# Figma to Code - 设计稿转代码

## 核心目标

将 Figma 设计稿高效、准确地转换为高质量的前端代码：
1. 保持设计稿的视觉效果
2. 复用现有组件和设计系统
3. 生成可维护的代码结构
4. 遵循项目编码规范

## 使用场景

### 必须调用的场景
- 用户发送 Figma 设计稿链接
- 需要根据设计稿实现页面
- 需要提取设计稿中的组件样式
- 设计稿与代码实现有偏差需要调整

### 不需要调用的场景
- 纯创意设计，无需编码
- 设计稿不完整或草稿状态
- 已有现成代码只需微调

## 工作流程

### 第一阶段：设计稿分析

1. **获取设计稿**
   - Figma 链接
   - 具体页面/Frame
   - 需要实现的范围

2. **分析设计系统**
   - 颜色规范（Color Palette）
   - 字体规范（Typography）
   - 间距规范（Spacing）
   - 组件库（Components）

3. **识别组件结构**
   - 可复用的基础组件
   - 页面级组件
   - 布局结构

### 第二阶段：技术映射

1. **框架选择**
   ```
   微信小程序 → WXML + WXSS
   React/Vue → 对应组件框架
   ```

2. **组件映射**
   ```
   Figma Component → 代码组件
   Auto Layout → Flexbox
   Constraints → 响应式布局
   ```

3. **样式提取**
   ```
   - 颜色：提取为 CSS 变量/主题色
   - 字体：提取为 typography 配置
   - 间距：提取为 spacing scale
   ```

### 第三阶段：代码生成

1. **文件结构规划**
   ```
   pages/
   ├── page-name/
   │   ├── page-name.wxml    # 结构
   │   ├── page-name.wxss    # 样式
   │   ├── page-name.js      # 逻辑
   │   └── page-name.json    # 配置
   components/
   ├── component-name/
   │   └── ...
   ```

2. **组件实现顺序**
   - 基础组件（按钮、输入框等）
   - 布局组件（卡片、列表等）
   - 页面组件

3. **样式实现**
   - 使用项目现有样式规范
   - 提取设计 token
   - 保持响应式

## 关键技术点

### 1. 微信小程序适配

```css
/* 使用 rpx 单位实现响应式 */
.container {
  width: 750rpx; /* 满屏宽度 */
  padding: 32rpx; /* 标准间距 */
}

/* 安全区域适配 */
.safe-area {
  padding-bottom: constant(safe-area-inset-bottom);
  padding-bottom: env(safe-area-inset-bottom);
}
```

### 2. 设计 Token 提取

```javascript
// utils/theme.js
module.exports = {
  colors: {
    primary: '#EAE8DC',
    secondary: '#3B8B9B',
    text: '#3A3A3A'
  },
  spacing: {
    xs: '8rpx',
    sm: '16rpx',
    md: '24rpx',
    lg: '32rpx'
  }
};
```

### 3. 组件化思维

```xml
<!-- 不好的做法：所有代码写在一起 -->
<view class="page">
  <view class="header">...</view>
  <view class="content">...</view>
  <view class="footer">...</view>
</view>

<!-- 好的做法：组件拆分 -->
<page-header title="{{title}}" />
<content-section list="{{list}}" />
<page-footer />
```

## 设计稿常见问题

### 1. 设计稿不完整
**问题**: 缺少状态、边界情况
**解决**: 
- 主动询问设计师
- 参考行业通用做法
- 记录待确认问题

### 2. 设计系统不统一
**问题**: 颜色、字体不一致
**解决**:
- 提取设计 Token
- 建立统一规范
- 逐步重构

### 3. 过度设计
**问题**: 效果复杂，实现成本高
**解决**:
- 评估 ROI
- 提供简化方案
- 分阶段实现

## 与设计师协作

### 需要确认的问题
1. 颜色值（特别是透明度）
2. 字体大小和行高
3. 间距尺寸
4. 交互状态（hover、active、disabled）
5. 边界情况（空状态、超长文本）

### 交付物
- 设计稿标注
- 切图资源
- 动效参数
- 设计规范文档

## 示例对话

**用户**: "这个页面按照 Figma 设计稿实现一下"

**你**: 
"好的，我先分析一下设计稿：

1. **设计系统提取**
   - 主色：#EAE8DC
   - 字体：14px/18px
   - 间距：8/16/24px

2. **组件拆解**
   - 顶部导航（已有）
   - 内容卡片（新增）
   - 底部按钮（已有）

3. **实现计划**
   - 第一步：搭建页面结构
   - 第二步：实现新增组件
   - 第三步：调整样式细节

有几个问题需要确认：
1. 这个状态的设计稿有吗？
2. 超长文本如何处理？
3. 动画效果需要实现吗？

我先实现基础版本，然后我们再一起调整细节。"

## 注意事项

1. **先整体后局部** - 先搭框架再扣细节
2. **保持沟通** - 有疑问及时确认
3. **渐进式实现** - 先完成再完美
4. **文档化** - 记录设计决策
