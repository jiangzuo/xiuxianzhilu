# 微信小程序编程规范

## 1. 代码风格

### 1.1 缩进与空格

- 使用 **2个空格** 作为缩进（与项目现有代码保持一致）
- 运算符两侧各加一个空格
- 逗号后加一个空格
- 花括号使用与语句同一行的风格

### 1.2 命名规范

- **变量名**：使用小驼峰命名法（如 `userInfo`、`currentHelpIndex`）
- **函数名**：使用小驼峰命名法（如 `refreshData`、`onShowHelp`）
- **常量名**：使用全大写蛇形命名法（如 `QUOTES_LIBRARY`）
- **组件名**：使用短横线分隔命名法（如 `custom-dialog`）
- **文件命名**：使用小写字母，单词间用短横线分隔（如 `profile.js`）

### 1.3 注释规范

- 使用 `//` 进行单行注释
- 关键逻辑和复杂代码块使用多行注释
- 注释应简洁明了，说明代码的目的和实现思路
- 函数和方法应添加注释说明其功能、参数和返回值


## 2. 页面开发规范

### 2.1 页面结构

- 每个页面应包含 `page.js`、`page.json`、`page.wxml`、`page.wxss` 四个文件
- 页面逻辑应在 `page.js` 中实现
- 页面配置应在 `page.json` 中定义
- 页面结构应在 `page.wxml` 中编写
- 页面样式应在 `page.wxss` 中定义

### 2.2 页面生命周期

- 正确使用页面生命周期函数：`onLoad`、`onShow`、`onReady`、`onHide`、`onUnload`
- 在 `onLoad` 中初始化页面数据和配置
- 在 `onShow` 中更新页面数据
- 在 `onUnload` 中清理定时器和监听器

### 2.3 数据管理

- 使用 `data` 对象存储页面数据
- 使用 `setData` 方法更新页面数据
- 避免频繁调用 `setData`，尽量合并更新
- 对于复杂数据，考虑使用缓存或全局数据管理

## 3. 组件开发规范

### 3.1 组件结构

- 每个组件应包含 `component.js`、`component.json`、`component.wxml`、`component.wxss` 四个文件
- 组件逻辑应在 `component.js` 中实现
- 组件配置应在 `component.json` 中定义
- 组件结构应在 `component.wxml` 中编写
- 组件样式应在 `component.wxss` 中定义

## 4. 服务与工具类规范

### 4.1 服务类

- 服务类应放在 `services` 目录下
- 服务类文件名使用 `.service.js` 后缀
- 服务类应封装特定功能的业务逻辑
- 服务类方法应返回Promise或回调函数

### 4.2 工具类

- 工具类应放在 `utils` 目录下
- 工具类文件名使用 `.js` 后缀
- 工具类应提供通用的工具函数
- 工具函数应简洁高效，专注于单一功能

## 5. 样式规范

### 5.1 样式文件

- 全局样式应在 `app.wxss` 中定义
- 页面样式应在对应页面的 `.wxss` 文件中定义
- 组件样式应在对应组件的 `.wxss` 文件中定义
- 可复用样式应在 `styles` 目录下定义

### 5.2 样式命名

- 样式类名使用短横线分隔命名法（如 `page-class`、`help-images`）
- 避免使用ID选择器
- 避免使用内联样式
- 样式应具有语义化，反映元素的功能

### 5.3 样式单位

- 使用 `rpx` 作为主要长度单位，适配不同屏幕尺寸
- 字体大小使用 `rpx` 单位
- 间距和边距使用 `rpx` 单位

## 6. 性能优化

### 6.1 代码优化

- 避免使用不必要的全局变量
- 避免在循环中使用 `setData`
- 避免频繁创建和销毁对象
- 使用 `wx.createSelectorQuery` 时应及时销毁

### 6.2 网络请求优化

- 合理使用缓存，减少网络请求
- 合并网络请求，减少请求次数
- 优化请求参数，减少数据传输量
- 使用 `wx.request` 的 `header` 配置，提高请求效率

### 6.3 渲染优化

- 避免使用复杂的 `wx:for` 循环
- 避免在模板中进行复杂的计算
- 使用 `hidden` 或 `wx:if` 条件渲染，减少 DOM 节点
- 合理使用 `scroll-view`，避免页面卡顿

## 7. 错误处理

### 7.1 异常捕获

- 使用 `try-catch` 捕获可能的异常
- 对网络请求和异步操作进行错误处理
- 对用户输入进行验证，防止非法输入

### 7.2 错误提示

- 使用 `wx.showToast` 显示错误提示
- 错误提示应清晰明了，告知用户具体错误原因
- 对于重要操作，应使用 `wx.showModal` 进行确认

## 8. 版本控制

### 8.1 Git 规范

- 提交代码前应检查代码格式和质量
- 提交信息应清晰明了，说明修改内容
- 避免提交不必要的文件（如 `node_modules`、`dist` 等）
- 使用 `.gitignore` 文件排除不需要版本控制的文件

### 8.2 代码审查

- 提交代码前应进行自我审查
- 团队成员应定期进行代码审查
- 代码审查应关注代码质量、性能和安全性

## 9. 安全规范

### 9.1 数据安全

- 敏感数据应进行加密存储
- 避免在客户端存储敏感信息
- 网络请求应使用 HTTPS 协议

### 9.2 代码安全

- 避免使用 `eval` 等危险函数
- 避免拼接 SQL 语句，防止 SQL 注入
- 避免使用 `setTimeout` 和 `setInterval` 导致的内存泄漏

## 10. 最佳实践

### 10.1 代码组织

- 按功能模块组织代码
- 使用模块化思想，将代码拆分为多个文件
- 合理使用 `require` 和 `module.exports` 进行模块导入和导出

### 10.2 命名约定

- 变量和函数命名应具有语义化
- 避免使用缩写和不明确的命名
- 保持命名风格一致

### 10.3 代码复用

- 提取重复代码为公共函数
- 使用组件化思想，复用UI组件
- 封装通用逻辑为服务类

### 10.4 文档规范

- 为重要的函数和模块添加文档注释
- 保持文档与代码同步更新
- 项目根目录应包含 README.md 文件，说明项目结构和使用方法
